How the UCC pipeline works

For demos: scrapers → filings → MCA leads → enrichment → compliance → export

How scrapers work

  • Each U.S. state (where implemented) has its own scraper: simple states use HTTP; harder portals (e.g. California behind a WAF) use Playwright.
  • The scheduler (when enabled on the pipeline service) runs jobs on a tier-based interval (e.g. Tier 1 about daily, Tier 2/3 less often).
  • Raw rows land in Postgres as ucc_filings. Inserts are deduplicated by state + filing_number so re-scrapes do not duplicate the same filing.
  • California searches by a set of secured-party keywords and a rolling filing-date window (configurable via SCRAPER_FILING_LOOKBACK_DAYS in .env). Results are de-duped in-memory across search terms before save.

Pipeline steps (conceptual order)

  1. Scrape — SOS portals → ucc_filings.
  2. MCA detection & scoring — For each filing, match secured party / collateral against mca_aliases and heuristics; scored rows become leads.
  3. Enrichment — Waterfall of vendors fills phone, email, etc.; cache + circuit breaker protect cost and stability.
  4. Compliance — Four ordered gates (internal DNC, DataMerch, DNC scrub, blacklist). Fail → lead blocked; pass → cleared.
  5. Export — Cleared leads go to GoHighLevel or CSV via a single adapter interface.
  6. Cross-state dedup (product target) — Exact + fuzzy matching before or as part of export (see roadmap).

Why many filings but zero leads?

Either MCA matching did not fire on those secured parties (seed aliases + heuristics), or the batch that turns filings into leads has not been run continuously in the process you are demoing.
